## 1. 架构设计评估

### 1.1 整体架构评价

#### 架构合理性评分

| 评估维度 | 评分 | 说明 |
|---------|------|------|
| 分层清晰度 | ★★★★★ | 用户层→接入层→服务层→数据层→外部依赖，层次划分清晰 |
| 职责分离 | ★★★★☆ | 各层职责明确，爬虫服务与主服务的数据耦合需关注 |
| 可扩展性 | ★★★★☆ | 模块化设计支持水平扩展，缺少服务发现机制 |
| 高可用设计 | ★★★☆☆ | 单点部署架构，缺少主从/集群方案 |
| 运维友好性 | ★★★★☆ | 阿里云生态集成度高，建议后续容器化 |

#### 当前架构全景

```
┌─────────────────────────────────────────────────────────────────┐
│  当前架构设计                                                    │
├─────────────────────────────────────────────────────────────────┤
│  微信小程序 ←→ Nginx ←→ NestJS API服务 ←→ MySQL/Redis           │
│                              ↓                                  │
│                        Python Scrapy 爬虫服务                   │
│                              ↓                                  │
│                         DeepSeek AI API                         │
└─────────────────────────────────────────────────────────────────┘
```

**架构优势**:
- ✅ 前后端分离，职责清晰
- ✅ 爬虫服务独立部署，不影响主业务
- ✅ 缓存层设计合理，减轻数据库压力
- ✅ NestJS模块化架构，代码组织清晰

**架构待改进**:
- ⚠️ 爬虫服务与API服务通过数据库耦合，缺少消息队列解耦
- ⚠️ 单节点部署存在单点故障风险
- ⚠️ 缺少服务注册/发现机制

### 1.2 模块划分评估

#### 模块职责矩阵

| 模块 | 职责定义 | 内聚性 | 耦合度 | 评估 |
|------|---------|--------|--------|------|
| **AuthModule** | 微信登录、Token管理 | 高 | 低 | ✅ 职责单一，设计合理 |
| **UserModule** | 用户信息管理 | 高 | 低 | ✅ 符合单一职责原则 |
| **UniversityModule** | 院校数据查询 | 高 | 中 | ✅ 设计合理 |
| **MajorModule** | 专业数据查询 | 高 | 中 | ✅ 设计合理 |
| **CampModule** | 夏令营CRUD、筛选 | 中 | 高 | ⚠️ 与UserSelection耦合，需关注 |
| **ReminderModule** | 提醒创建/推送 | 中 | 高 | ⚠️ 与微信API强耦合，需熔断机制 |

#### 模块依赖关系

```
AuthModule
    ↓
UserModule ←──────┐
    ↓             │
UniversityModule ─┼──→ CampModule ←── UserSelection
    ↓             │       ↓
MajorModule ──────┘    ReminderModule
```

**问题识别**:
- CampModule与UserSelection存在循环依赖风险
- ReminderModule与微信服务强耦合

**改进建议**:
1. 引入防腐层(Anti-Corruption Layer)隔离微信API
2. 使用事件驱动模式解耦模块间通信

## 2. 技术选型评估

### 2.1 技术栈评估矩阵

| 技术组件 | 选型 | 适用性 | 风险等级 | 建议 |
|---------|------|--------|---------|------|
| **后端框架** | NestJS 10.x | ★★★★★ | 低 | ✅ TypeScript全栈，模块化优秀，强烈推荐 |
| **ORM** | Prisma 5.x | ★★★★★ | 低 | ✅ 类型安全，迁移管理方便，推荐 |
| **数据库** | MySQL 8.0 | ★★★★☆ | 低 | ✅ 数据量可控，运维成熟，适合MVP |
| **缓存** | Redis 7.0 | ★★★★★ | 低 | ✅ 功能丰富，支持队列和分布式锁 |
| **爬虫框架** | Scrapy 2.x | ★★★★★ | 中 | ✅ 业界标准，需独立维护Python环境 |
| **AI提取** | DeepSeek API | ★★★★☆ | 中 | ✅ 性价比高，需关注API稳定性 |
| **部署** | ECS+PM2 | ★★★☆☆ | 中 | ⚠️ 传统部署，建议后续容器化 |

### 2.2 关键选型分析

#### NestJS 选型理由

**优势**:
- TypeScript全栈，前后端类型可共享
- 依赖注入、装饰器语法，代码结构清晰
- 内置验证、拦截器、管道等功能
- 与Prisma集成良好

**考量**:
- 学习曲线较陡，团队需要上手时间
- 运行时性能略低于原生Express（差异可忽略）

#### Scrapy + DeepSeek 组合

**优势**:
- Scrapy是爬虫领域最成熟的框架
- DeepSeek中文理解能力强，性价比高
- AI辅助提取可大幅提升信息结构化准确率

**风险与缓解**:

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| DeepSeek API不稳定 | 信息提取失败 | 设计降级方案：正则提取兜底 |
| 爬取频率限制 | 数据更新延迟 | 配置代理池，控制请求频率 |
| 院校官网反爬 | 数据源失效 | 多数据源备份+人工兜底 |

```typescript

  secret: process.env.JWT_SECRET,      // 强密钥，环境变量存储
```

```typescript
// 建议的限流配置
const rateLimitConfig = {
  global: { ttl: 60, limit: 1000 },    // 全局限流：每分钟1000请求

  login: { ttl: 300, limit: 5 },       // 登录接口：每5分钟5次

```

## 6. 风险评估

### 6.1 致命风险（必须立即处理）

#### R1: 爬虫法律合规风险

| 项目 | 内容 |
|------|------|
| **风险描述** | 自动采集高校官网数据可能违反《反不正当竞争法》和院校服务条款 |
| **发生概率** | 中 |
| **影响程度** | 致命 |
| **风险等级** | 🔴 **致命** |

**应对措施**:
1. ⚠️ **立即咨询法律顾问**，确认爬虫合规性
2. 控制爬取频率（每个域名≤1请求/分钟）
3. 遵守robots.txt规则
4. 准备免责声明和用户协议
5. 建立院校投诉处理机制

#### R2: DeepSeek API成本控制 ✅ 已解决

| 项目 | 内容 |
|------|------|
| **风险描述** | 未设置调用上限，月度费用可能暴涨 |
| **发生概率** | 低（已配置动态扩缩容） |
| **影响程度** | 中 |
| **风险等级** | 🟢 **已控制** |

**已实施措施**:
1. ✅ **动态扩缩容配置**: 基础200次/天，峰值500次/天，上限800次/天
2. ✅ **优先级队列**: P0(Top7)立即处理，P1(985)1小时内，P2(211)2小时内，P3(其他)4小时内
3. ✅ **成本控制**: 日预算50元，月预算1000元，超80%触发告警
4. ✅ **熔断机制**: 连续失败10次暂停1小时

**成本预估**:
| 场景 | 日均调用 | 日均成本 | 月均成本 |
|------|----------|----------|----------|
| 日常运营 | 200次 | 20元 | 600元 |
| 高峰期 | 500次 | 50元 | 1500元 |
| 极端峰值 | 800次 | 80元 | 2400元 |

> **配置依据**: 以每日接收免试研究生的院校所发布的夏令营公告数量为基准（366所院校，高峰期50-300条/天），设置尽可能高的API调用上限值。

#### R3: 开发周期评估 ✅ 已调整

| 项目 | 内容 |

| **风险描述** | 技术栈复杂度高，4周难以完成全部功能 |

## 10. 总结与建议

### 10.1 架构设计总体评价

**✅ 优势**:
1. 技术选型合理，符合MVP阶段需求
2. 模块划分清晰，职责单一
3. 数据流设计合理，缓存策略得当
4. NestJS+Prisma组合现代化，开发效率高
5. 安全设计基本完善

**⚠️ 待改进**:
1. 爬虫与主服务耦合度较高，建议引入消息队列
2. 单点部署存在风险，建议后续容器化
3. 部分数据库索引需要优化
4. 缺少完善的监控告警体系
5. 4周开发周期偏紧，建议调整

### 10.2 关键成功因素

1. **数据质量**: 爬虫+AI提取的准确率是产品核心竞争力
2. **提醒触达**: 微信订阅消息的稳定性直接影响用户体验
3. **性能保障**: 高峰期（截止日前）的系统稳定性
4. **法律合规**: 爬虫合规性是项目存续的基础
